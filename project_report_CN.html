2. Problem Statement: The DPI Blind Spot

üö® The Challenge: Deep Packet Inspection (DPI) Failure

  • 90%+ of web traffic is encrypted (HTTPS/TLS), rendering traditional Deep Packet Inspection (DPI) useless.
  • Signature-based IDSs are inherently blind to Zero-Day attacks (novel threats).
  • Privacy vs. Security: Decrypting traffic (MITM) introduces latency and violates privacy (GDPR).
  • Complex Attacks: Modern threats use stealth and behavioral patterns that evade simple rule-sets.

‚úÖ Our Solution: Encryption-Agnostic Flow Analysis

  • Flow-Based Analysis: Analyze only aggregated metadata (packet size, timing, flags), which are visible even when the payload is encrypted.
  • AI-Powered Detection: XGBoost detects complex behavioral patterns and statistical anomalies.
  • Privacy-Preserving: No payload data is ever required or decrypted.
  • Zero-Day Protection: Anomaly detection catches novel attacks by flagging flows that deviate from normal network behavior.

3. Flow Metadata & Behavioral Detection

üí° The Core Concept: Flow Features

Instead of inspecting payload content (Layer 7), we analyze statistical aggregates derived from packet headers (Layers 3 & 4) over the duration of a session.

What is a Flow? (The 5-Tuple)

A flow is a sequence of packets sharing the same 5-tuple: Source IP, Destination IP, Source Port, Destination Port, and Protocol (TCP/UDP). This aggregation abstracts individual packets into a single, comprehensive record of network behavior.

Key Features We Extract from Flows

üìä Timing & Rate Features

Measures like Flow IAT (Inter-Arrival Time) Mean/Std, Flow Duration, and Packets/Second. Crucial for identifying DDoS (high rate, low IAT) and Botnet C&C (periodic, low IAT Std).

üì¶ Packet Size Statistics

Statistics on packet lengths: Mean, Max, Min, Standard Deviation, and Forward/Backward Ratio. Used to detect Heartbleed (response >> request) and Web Attacks (unusually long request packets).

üö© TCP Flags & Control (Layer 4 Detail)

Counts and states of control bits like SYN, ACK, FIN, RST, PSH, URG, and the Window Size. Essential for identifying PortScans (many SYN/RST without ACK) and specific TCP-based DoS attacks.

4. System Architecture: Three-Tiered Defense

Figure 1: Complete AI-IDS Pipeline: Packet Capture → Flow Extraction → Preprocessing → Tiered AI Classification

üîÑ The 3-Stage Inference Logic (Our Defense Tiers)

Tier 1: The Gatekeeper (Binary Classification)

Role: High-speed initial screening to separate the overwhelming volume of benign traffic from malicious flows.

Algorithm: Optimized XGBoost Classifier | Objective: Maximize Recall (minimize False Negatives) to ensure all threats are flagged.

Performance: Accuracy ~99.9%, AUC 1.00.

Tier 2: The Diagnostician (Multiclass Classification)

Role: Deep inspection of flows flagged as malicious by the Gatekeeper, providing threat specificity.

Algorithm: XGBoost (Multiclass) | Output: Identifies 14 specific attack types (DDoS, Botnet, PortScan, etc.) based on their unique flow feature signatures.

Performance: Accuracy ~97.5%, providing high confidence attack type identification.

Tier 3: The Safety Net (Anomaly Detection)

Role: Zero-Day/Novel Threat Detection. Catches attacks not present in the training data.

Algorithm: Deep Autoencoder | Mechanism: Trained ONLY on benign traffic. An abnormally high Reconstruction Error (MSE) flags a flow as an anomaly (i.e., potential novel attack).

5. Methodology & Data Engineering Pipeline

A meticulous pipeline was required to transform raw PCAPs into optimized, machine-learning-ready flow features.

Data Acquisition and Initial Processing

PCAP Capture & Flow Aggregation

Dataset: CICIDS2017 (~2.8M flows). Tool: CICFlowMeter used to generate 77 statistical features from raw packet captures (PCAP files).

Pre-processing and Feature Engineering (The Critical Steps)

1. Data Cleaning & Downsampling

Handling of ∞ values, dropping NaNs, and removing network identifiers (IPs, Ports). Benign flows were downsampled to 1.2√ó the attack count to manage severe class imbalance.

2. Standard Scaling

Z-score normalization was applied to all 77 features. This standardizes features to Œº = 0 and œÉ = 1, which is crucial for algorithms reliant on distance metrics (like K-Means and PCA).

3. K-Means Clustering (Feature Engineering)

Cluster labels (6 for binary, 10 for multiclass) were generated and added as a new, engineered feature. This helps the model separate attacks with similar low-level statistics.

4. PCA Dimensionality Reduction

Reduced the 78 features (77 original + 1 cluster ID) to 20 Principal Components (PCs). This minimizes computational load and overfitting risk while retaining over 95% of the data variance.

Attack Detection Patterns (Behavioral Fingerprinting)

DoS / DDoS

Fingerprint: Extremely low Flow Inter-Arrival Time (IAT), coupled with high Packets/s and specific TCP flag combinations.

PortScan

Fingerprint: Many short flows (low Total Fwd Packets), high count of flows with only SYN flags set (unsuccessful connection attempts).

Botnet

Fingerprint: Machine-like communication indicated by very low Flow IAT Standard Deviation (periodic C&C beacons) and highly predictable Packet Length Means.

Heartbleed

Fingerprint: Asymmetry: Small request packet → Huge response packet. High Bwd Packet Length Max (signifying a large response size).

Web Attacks (XSS/SQLi)

Fingerprint: Large Payloads: Fwd Packet Length Max is higher than normal web requests due to long SQL strings or scripts being injected into parameters.

Infiltration

Fingerprint: Upload Heavy: Subflow Fwd Bytes (Data leaving network) → Subflow Bwd Bytes (Data entering), indicating mass data exfiltration.

9. Challenges & Future Scope

üõ†Ô∏è Major Technical Challenges

Challenge 1: Training-Testing Mismatch (Domain Shift)

The feature distribution of live data (home/campus WiFi, different IPs/OSes) varied significantly from the controlled, lab-captured data (CICIDS2017). This "Domain Shift" caused the model to struggle with generalization.

Challenge 2: Missing Features & Data Integrity

Crucial features (like Source/Destination Port) were found to be missing or zeroed out in live flow extraction due to tool/network limitations. This directly impacted detection logic, as essential context (e.g., knowing a port is 80 or 443) was lost.

Challenge 3: Threshold Sensitivity (Autoencoder)

The fixed anomaly threshold, calculated during training, often led to high False Positive rates in production. The solution required exploring dynamic and percentile-based thresholds to adapt to the shifting definition of "normal" behavior in live traffic.

Challenge 4: Reproducibility Issues

Tiny variances in floating point precision (e.g., between macOS and Windows) or library version differences resulted in inconsistent prediction scores near the decision boundary. The solution involved using Docker for environment control and setting explicit random seeds.

Challenge 5: Live Traffic Characteristics

Due to the rarity of attacks in real-world scenarios and ethical constraints, testing the IDS required specific workarounds, primarily replaying labeled attack PCAPs in isolated test environments.

Future Scope: Continuous Improvement

  • Online Learning: Implement incremental model updates from live traffic to adapt to evolving threats without full retraining.
  • Temporal Analysis: Integrate models like LSTM/CNN to analyze time-series dependencies within flows, adding context to the simple statistical features.
  • Deployment: Move from a single API endpoint to a distributed streaming architecture (e.g., Kafka + Flink/Spark Streaming) for true high-volume, real-time operation.

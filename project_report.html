1. Introduction & Motivation

‚ùå The Problem

  • 90%+ of web traffic is encrypted (HTTPS), rendering traditional Deep Packet Inspection (DPI) useless
  • Traditional firewalls rely on "Signatures" (databases of known bad IPs), making them blind to Zero-Day attacks (new, unknown threats)
  • Signature-based systems cannot detect novel attack patterns

‚úÖ Our Solution

  • Flow-Based Analysis: We analyze metadata (packet size, timing, flags), not encrypted content
  • AI-Powered: Machine Learning detects complex patterns human rules miss
  • Privacy-Preserving: No user data is ever decrypted

2. The Dataset (CICIDS2017)

üìä Dataset Overview

  • Source: CICIDS2017 (Canadian Institute for Cybersecurity)
  • Scale: ~2.8 Million Network Flows
  • Classes: 15 Total (1 Benign + 14 Attack Types)
  • Features: 80+ statistical features (e.g., Flow Duration, Total Fwd Packets, Fwd Packet Length Max)
  • Processed: Reduced to 20 Principal Components (PCA) for efficiency, plus K-Means Cluster IDs

üîç Feature-to-Attack Detection Matrix

How we detect different attack types using network flow patterns:

Attack Type Network Pattern Critical Features
DoS / DDoS "Flooding the server" High Packet Rate (Flow Packets/s), Small Inter-Arrival Time (Flow IAT), High Congestion Flags (CWE/ECE Flag)
Heartbleed "Leaking memory" Asymmetry: Small request packet ‚Üí Huge response packet. High Bwd Packet Length Max
Botnet (ARES) "Robotic consistency" Low Variance: Flow IAT Std is very low (machine-like periodic beacons), distinct Packet Length Mean
PortScan "Knocking on doors" Short & Tiny: Thousands of flows with Total Packets = 2-3, SYN Flag = 1, very short Flow Duration
Web Attacks (XSS/SQLi) "Injection payloads" Large Payloads: Fwd Packet Length Max is higher than normal web requests (due to long SQL strings or scripts)
Infiltration "Data theft" Upload Heavy: Subflow Fwd Bytes (Data leaving network) > Subflow Bwd Bytes (Data entering)

5. Phase 2 - Final System Architecture

üîÑ System Pipeline

Figure 4: Complete System Pipeline - From PCAP Input to Attack Classification

üéØ The 3-Stage Inference Logic

Stage 1: Gatekeeper (Binary XGBoost)

Question: Is this traffic Benign or Malicious?

Result: If Benign ‚Üí Check Anomaly. If Attack ‚Üí Go to Stage 2.

Performance: Accuracy ~99.9%, AUC 1.00

Stage 2: Diagnostician (Multiclass XGBoost)

Question: What specific attack is this?

Result: Returns "DDoS", "Botnet", etc. (Only if Confidence > 60%)

Capabilities: Detects 14 classes (Bot, DDoS, GoldenEye, Hulk, Slowloris, FTP-Patator, Heartbleed, etc.)

Performance: Accuracy ~97.5%, Micro-Averaged AUC: Near 1.0

Stage 3: Safety Net (Autoencoder Anomaly Detector)

Question: Does this match "Normal" traffic patterns?

Mechanism: A Neural Network trains to reconstruct normal traffic. If Reconstruction Error (MSE) is high, it's an ANOMALY.

Why: Catches Zero-Day attacks the XGBoost models haven't seen before

Training Strategy: Trained ONLY on Normal (Benign) traffic

6. Data Engineering & Preprocessing

Comprehensive preprocessing pipeline implemented in final_data_preprocess.py to handle the complex CICIDS2017 dataset:

üìã Key Techniques

1. Dataset Loading & Cleaning

Load all CSV files from TrafficLabelling folder, combine into single DataFrame. Remove non-numeric columns (Flow ID, IPs, timestamps, ports, protocol). Handle infinite values, drop NaN and duplicates. Final dataset: ~2.8M flows with 77 numeric features.

2. Label Generation

Create Binary_Label (0=Benign, 1=Attack) and Multiclass_Label using LabelEncoder for 14 attack types. Save label encoder for inference consistency.

3. Benign Reduction

Downsample benign traffic to 1.2x attack samples (target_benign = len(attack_df) * 1.2) to prevent model bias while maintaining sufficient normal samples for training.

4. Standard Scaling

Apply Z-score normalization using StandardScaler. Fit on training DataFrame to preserve feature_names_in_ attribute. Transform all features to mean=0, std=1. Save scaler.pkl and feature_names.pkl for inference.

5. Class Balancing

Binary: RandomOverSampler to balance classes. Multiclass: Cap each class at 60,000 samples (no oversampling) to maintain natural distribution. Anomaly: Extract only benign samples, sample down to 80,000 if larger.

6. K-Means Clustering

Add cluster labels as additional features: 6 clusters for binary, 10 clusters for multiclass. Helps separate hard-to-distinguish attacks. Save kmeans_bin.pkl and kmeans_multi.pkl models.

7. PCA Dimensionality Reduction

Reduce features to 20 principal components: 77 numeric + 1 cluster = 78 features ‚Üí 20 PCs. Preserves 95%+ variance, reduces overfitting, speeds up training. Separate PCA models for binary (pca_bin.pkl) and multiclass (pca_multi.pkl).

8. Train/Test Split

80/20 stratified split ensuring balanced class distribution across train and test sets. Save as CSV files: binary_train.csv, binary_test.csv, multiclass_train.csv, multiclass_test.csv, anomaly_train.csv, anomaly_test.csv.

7. Model Details

7.1 Binary Classification (The Gatekeeper)

Algorithm: XGBoost Classifier | Accuracy: 99.9% | AUC: 1.00

Configuration: 250 trees, depth 8, learning rate 0.08, 80% subsampling

Key Insight: Packet Length Std and Flow IAT are top predictors of attacks.

7.2 Multiclass Classification (The Diagnostician)

Algorithm: XGBoost Classifier (multiclass) | Accuracy: 97.5% | Micro-AUC: Near 1.0

Configuration: 300 trees, depth 10, learning rate 0.08 | 14 Attack Classes: Bot, DDoS, DoS variants, FTP/SSH-Patator, Heartbleed, Infiltration, PortScan, Web Attacks

Detection Patterns: DDoS (high packet rates), Heartbleed (small request ‚Üí huge response), Botnet (periodic timing), PortScan (short flows with SYN flags)

7.3 Anomaly Detection (The Safety Net)

Model: Deep Autoencoder (Neural Network) | Training: Only on normal traffic (80k samples)

Architecture: Input 20 ‚Üí Encoder (32‚Üí16‚Üí10) ‚Üí Decoder (16‚Üí32‚Üí20) | Loss: MSE | Threshold: mean + 3√óstd

Purpose: Detects zero-day attacks and novel threats by learning normal patterns and flagging high reconstruction errors.
